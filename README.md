# on-my-interview

一个面向面经内容沉淀的智能采集与结构化分析系统。它把分散在社区里的原始面经帖子，转换成可检索、可统计、可持续积累的结构化资产，服务于题库建设、知识图谱、候选人研究和招聘情报分析。

## 这个项目解决什么问题

面经帖子天然是非结构化内容，通常分散在不同平台、描述风格不统一、重复内容多、可复用性差。团队即使收集到了大量帖子，也很难稳定回答这些业务问题：

- 某家公司最近常问哪些题
- 某个岗位常出现哪些技术标签
- 一篇帖子里有哪些关键面试事件和情绪信号
- 哪些原始帖子已经抓过，哪些是增量
- 当前结构化解析大约花了多少 LLM 成本

`on-my-interview` 的目标，就是把“帖子”变成“数据资产”。

## 核心能力

- 自动抓取面经帖子，并保留原始内容和源站元数据
- 以 `(platform, source_post_id)` 为主键做去重，避免重复抓取和重复解析
- 通过可替换的 LLM 抽象接口，把原始帖子转换成结构化 JSON
- 从正文中识别公司、情绪、关键事件和面试题
- 按“最小行原则”拆分面试题，一题一行，并为每题打上可扩展标签
- 将题目和标签落库，为后续知识图谱、题库检索和统计分析提供基础数据
- 对 LLM usage 做北京时间 12 小时窗口聚合，便于追踪总 token 和估算花费

## 结构化结果示意

下面是系统处理一篇面经后，业务上真正可消费的结果形态示意：

```json
{
  "platform": "nowcoder",
  "company": "美团",
  "sentiment": "positive",
  "key_events": [
    "一面偏项目深挖",
    "二面追问并发和缓存一致性"
  ],
  "questions": [
    {
      "question": "Redis 和 MySQL 双写时如何保证一致性？",
      "tags": ["后端", "Redis", "一致性"]
    },
    {
      "question": "Kafka 为什么会出现消息重复消费？",
      "tags": ["中间件", "Kafka", "MQ"]
    }
  ]
}
```

这类结果可以直接用于：

- 公司题库沉淀
- 技术标签聚合
- 面试重点事件分析
- 知识图谱构建
- 面经数据质量评估

## 业务流程

```text
社区面经帖子
  -> 抓取原帖
  -> MySQL 去重和原始落库
  -> LLM 结构化解析
  -> 题目最小行拆分与标签化
  -> 结构化查询 / 公司聚合 / 成本统计
```

这条链路把“内容采集”升级成了“面试知识生产线”。

## 系统架构

当前仓库是一个后端优先、API 驱动的实现，核心结构如下：

- `server/`
  - Go 服务主体，负责抓取、调度、解析、落库和查询 API
- `docker-compose.dev.yml`
  - 本地开发依赖，当前包含 MySQL 和 Adminer
- `docs/plans/`
  - 每次任务的上下文、设计和实现痕迹，保证研发过程可追踪
- `memory/`
  - 项目的长期事实、经验和决策沉淀
- `rules/`
  - 仓库级研发规则和工作方式约束

从能力边界看，系统内部已经拆成几层：

- 平台抓取层：当前已实现牛客平台
- 存储层：负责原始帖子、结构化结果、题目、标签、任务和 usage 聚合
- LLM 抽取层：默认使用 OpenAI-compatible 接口，已兼容 DeepSeek 接入方式
- 服务接口层：提供健康检查、抓取任务、帖子查询、题目查询、公司聚合和 usage 查询

## 当前实现范围

当前版本已经落地的范围是：

- 已实现牛客面经抓取
- 已实现 MySQL 去重和原始帖子落库
- 已实现结构化解析和题目拆分入库
- 已实现标签拆分和公司聚合查询
- 已实现北京时间 12 小时 usage 聚合
- 已提供 HTTP 接口和定时任务能力

当前还没有做的部分也很明确：

- 还没有完整前端产品界面
- 当前主要依赖搜索接口抓取，不是详情页全文抓取
- 多平台扩展架构已预留，但首个正式实现平台是牛客

## 适用场景

- 招聘团队沉淀公司面经题库
- 教培或求职辅导团队整理高频题和技术标签
- 数据团队构建面经知识图谱
- 对不同公司、岗位、技术方向做横向分析
- 追踪结构化抽取的 usage 和大致成本

## 快速开始

如果你想快速在本地跑起来：

```bash
cp .env.example .env
docker compose -f docker-compose.dev.yml up -d
cd server && go run ./cmd/server
```

默认本地可用入口：

- API: `http://localhost:8080`
- Adminer: `http://localhost:8081`

更完整的服务启动、环境变量和接口说明，见 [server/README.md](./server/README.md)。

## 项目定位

这不是一个单纯的“爬虫脚本”仓库。它更接近一个面经数据基础设施项目：前面连接内容来源，后面连接题库、图谱、分析和成本观测。当前版本已经把最关键的一条链路打通，后续可以继续向多平台、全文抓取、规则归一化和上层产品能力扩展。
